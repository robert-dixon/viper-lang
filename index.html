'Generates pythagorean triplets up to given side length'
pythag n =
    for z in 1 to n
        for y in 1 to n
            for x in 1 to y
                if x^2 + y^2 is z^2
                    [ x y z ]

"Please enter a maximum side length:"
pythag! toInt! read!

operate fn x y =
    fn! x y

add x y =
    return x + y

mult x y =
    return x * y

operate! add 3 2
operate! mult 3 2

newObj a b =
    obj = new
    obj.a = a
    obj.b = b
    obj.display =
        return this.a + this.b
    return obj

inst = newObj! 3 5
inst.display!

Why Viper?

Viper is yet another programming language. I didn't build it to solve any great unsolved programming challenges or facilitate the easy implementation of a niche design pattern. I built it because I was interested in programming language design, and now I've designed one - and it's just the way I like it. If you happen to like it too, the please do join in! It's there to be enjoyed.

Let me explain what it's for. I designed this language with hackathons in mind. It's a language for quickly throwing together little programs that behave nicely, not for constructing large architecturally complex or high-performant systems. And coding in it should be simple. You're brain shouldn't be hurting after a 14-hour stretch.

So, sure, it's not a ground-breaking earth-shattering development in programming language design, but it still does some things quite well. Working with higher-order functions is as nice in Viper as is possible for a procedural language - you can just throw them around like variables. Also it has a number of language features that are well-suited to its intended purpose: list structures that can contain multiple data types can be used as generic data objects and are very versitile and quick to use, dynamic object schemas means you don't have to spend ages designing your classes and inheritence structures when you're coding in a hurry, and so on.

There are a number of controvertial design decisions - I'd have it no other way. For example, you might question why I have strayed from convention in abandoning parantheses around function arguments. Truthfully, the answer is I like the look of this better: I feel that if you're tired or have been working for hours, looking at a page full of punctuation makes you less able to concentrate on the logic of your code - so let's eliminate as much of it as possible and set out the logic as visibly as possible in words and spacing. Maybe many moons ago, people needed the brackets to clearly distingish their arguments from their function names but as far as I'm concerned, syntax highlighting has solved that problem. It's not for everyone, but I guess that is one the perks of writing your own language: you can have it your way.
